# Review Checklist

이 문서는 "무엇을 점검할지"를 정의한다.
최종 severity 판정은 `team-rules.md`를 따른다.

## finding 확장 원칙

- 리뷰 결과를 만들 때 `blocking`만 찾고 끝내지 않는다.
- 아래 체크리스트를 보며 `suggestion`과 `question` 후보도 함께 수집한다.
- 특히 테스트 보강, 설명 보강, 샘플 설정, 경계 조건 처리, 의도 확인이 필요한 부분은 `P3`, `P4`, `question`으로 남길지 한 번 더 검토한다.

## 안정성

- null 또는 empty 입력 처리 누락이 없는지 확인한다.
- 예외 처리와 실패 경로가 명확한지 확인한다.
- 외부 API, 파일 업로드, DB 실패 시 복구 전략이나 사용자 피드백이 있는지 확인한다.
- 비동기 처리나 2단계 요청이 부분 성공 상태를 만들지 않는지 확인한다.
- 설정값 누락 시 애플리케이션 기동 또는 요청 처리에 실패하지 않는지 확인한다.

## 보안

- 인증과 인가 검증이 적절한지 확인한다.
- 민감 정보가 로그, 응답, 에러 메시지에 노출되지 않는지 확인한다.
- 사용자 입력 검증과 escaping이 필요한 부분이 누락되지 않았는지 확인한다.
- HTML 렌더링, 파일 업로드, 리치 텍스트 입력에서 XSS/스크립트 삽입 가능성이 없는지 확인한다.
- 허용 파일 형식, 크기, 개수 제한이 서버와 클라이언트에서 일관되는지 확인한다.

## 데이터 정합성

- 중복 저장, 누락 저장, 경쟁 상태 가능성이 없는지 확인한다.
- 수정/삭제 시 연관 데이터가 함께 올바르게 정리되는지 확인한다.
- 트랜잭션 경계가 적절한지 확인한다.
- migration이나 schema 변경이 역호환성을 해치지 않는지 확인한다.
- Mapper SQL과 Java 파라미터 이름이 정확히 매칭되는지 확인한다.

## API 및 계약

- 요청/응답 JSON 구조 변경이 기존 클라이언트와 충돌하지 않는지 확인한다.
- 상태 코드가 의도에 맞는지 확인한다.
- path variable, query parameter, request body 간 역할이 일관적인지 확인한다.
- 프론트엔드와 백엔드가 같은 유효성 규칙을 공유하는지 확인한다.

## UI 및 사용자 흐름

- 화면 렌더링에 필요한 모델 값이 빠지지 않았는지 확인한다.
- 등록/수정/삭제 이후 이동 경로가 올바른지 확인한다.
- 취소, 재시도, 오류 메시지 흐름이 사용자 관점에서 이해 가능한지 확인한다.
- 첨부파일, 이미지, 드래그앤드롭, 에디터 같은 상호작용에서 실패 케이스가 처리되는지 확인한다.

## 테스트

- 핵심 분기와 실패 경로를 검증하는 테스트가 있는지 확인한다.
- 회귀 가능성이 높은 부분에 보강 테스트가 필요한지 확인한다.
- 템플릿/모델 계약, mapper 파라미터, 설정 의존성 같은 회귀 포인트를 테스트가 덮는지 확인한다.
- merge 차단 사유는 아니어도 테스트 한두 개를 더 추가하면 회귀를 줄일 수 있는 지점을 `P3` 또는 `P4` 후보로 본다.

## 성능

- 반복 호출, N+1, 불필요한 전체 조회가 없는지 확인한다.
- 입력 규모가 커졌을 때 병목 가능성이 있는지 확인한다.
- 리스트 비교, 파일 처리, HTML 파싱이 큰 입력에서 비효율적으로 동작하지 않는지 확인한다.

## 운영 및 배포

- 신규 설정값, 환경 변수, 샘플 설정 파일 업데이트가 필요한지 확인한다.
- 배포 후 장애 시 롤백 가능한지 확인한다.
- CI, 파이프라인, 필수 검증 단계가 비어 있거나 누락되지 않았는지 확인한다.
- 운영 문서나 샘플 설정이 조금만 보강돼도 다음 배포가 쉬워지는 경우 `P3`, `P4` suggestion 후보로 본다.

## Open question 후보 체크

- 구현 의도가 코드만으로 충분히 드러나는지 확인한다.
- 권한 정책, 비즈니스 규칙, 배포 환경 정보가 없어서 단정하기 어려운 부분이 있는지 확인한다.
- 현재 분기 처리가 의도된 제한인지 임시 처리인지 질문이 필요한지 확인한다.
