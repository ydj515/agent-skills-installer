# Team Rules

이 문서는 "무엇을 blocking issue, non-blocking suggestion, open question으로 분류할지"를 정의한다.
기술 점검 항목 자체는 `review-checklist.md`를 우선 보고, 최종 판정은 이 문서를 따른다.
또한 리뷰 코멘트에는 강조 수준을 나타내는 `P1` ~ `P5` prefix를 사용한다.

## 판정 우선순위

1. 사용자 영향과 운영 리스크를 먼저 본다.
2. 그 다음 데이터 정합성과 보안을 본다.
3. 마지막으로 유지보수성과 테스트 보강 필요 여부를 본다.

## 등록 대상 원칙

- 리뷰는 `Blocking issue`만 찾는 작업이 아니다.
- merge 차단 사유가 없더라도, 가치 있는 `Non-blocking suggestion`과 `Open question`은 적극적으로 finding으로 남긴다.
- 특히 회귀 방지 테스트, 가독성 개선, 문서화, 설정 샘플 보강, 의도 확인이 필요한 분기점은 `P3`, `P4`, `question`으로 다루는 것을 기본 흐름으로 본다.
- 다만 코멘트 수를 늘리기 위한 억지 지적은 남기지 않는다. 중복이거나 가치가 낮으면 생략한다.

## Pn Prefix 규칙

리뷰 코멘트는 가능하면 제목 앞에 `[P1]`, `[P2]`, `[P3]`, `[P4]`, `[P5]` prefix를 붙여 강조 수준을 표현한다.
형식은 아래처럼 맞춘다.

```text
[P1] 설정 누락으로 애플리케이션이 부팅되지 않을 수 있습니다.
[P3] 테스트 케이스를 한두 개 더 보강하면 회귀를 줄이는 데 도움이 됩니다.
[P4] 이 분기는 주석으로 의도를 한 줄 남기면 다음 수정자가 이해하기 쉬워집니다.
```

### P1: 꼭 반영해주세요 (Request changes)

- 서비스에 중대한 오류를 발생시킬 수 있는 가능성이 있거나, merge 전에 반드시 수정이 필요한 경우 사용한다.
- 리뷰 요청자는 P1을 반영하거나, 반영할 수 없는 합리적인 근거로 리뷰어를 설득할 수 있어야 한다.
- 일반적으로 `Blocking issue`와 함께 사용한다.

### P2: 적극적으로 고려해주세요 (Request changes)

- merge 자체를 즉시 막을 정도는 아닐 수 있지만, 의미 있는 리스크나 유지보수 문제로 인해 적극적인 수정 논의가 필요한 경우 사용한다.
- 작성자는 P2를 수용하거나, 수용이 어렵다면 적절한 의견으로 토론하는 것이 권장된다.
- 상황에 따라 `Blocking issue` 또는 강한 `Non-blocking suggestion`으로 연결될 수 있다.

### P3: 웬만하면 반영해 주세요 (Comment)

- 개선 가치가 분명하고 되도록 반영되면 좋지만, 반드시 merge 전에 해결해야 하는 것은 아닌 경우 사용한다.
- 작성자는 P3을 수용하거나, 수용이 어렵다면 이유를 설명하거나 다음 반영 계획을 명시적으로 남기는 것이 권장된다.
- `Request changes` 가 아닌 `Comment` 성격의 제안에 사용한다.

### P4: 반영해도 좋고 넘어가도 좋습니다 (Approve)

- 반영하면 더 좋아질 수 있지만, 작성자가 답변하지 않고 넘어가도 괜찮은 수준의 의견에 사용한다.
- 작성자는 P4를 무시해도 괜찮다.
- 주로 가독성, 일관성, 작은 개선 아이디어, 설명 보강에 사용한다.

### P5: 그냥 사소한 의견입니다 (Approve)

- 정말 사소한 의견이나 취향 수준의 의견에 사용한다.
- 작성자는 P5에 대해 아무런 의견을 달지 않고 무시해도 괜찮다.

## Blocking issue 기준

아래 중 하나에 해당하면 기본적으로 `Blocking issue`로 분류한다.

- 운영 장애를 유발할 가능성이 높은 변경
- 인증 또는 인가 누락
- 민감 정보 노출 가능성
- 데이터 손실, 중복 저장, 잘못된 삭제 가능성
- 애플리케이션 부팅 실패, 배포 실패, 설정 누락 위험
- 핵심 사용자 플로우가 즉시 깨지는 UI/API/비즈니스 로직 문제
- 잘못된 SQL, mapper 바인딩 오류, transaction 누락 등으로 요청 자체가 실패할 가능성이 큰 문제
- 롤백 전략 없는 위험한 migration 또는 운영 설정 변경
- 핵심 플로우에 대해 merge 전 반드시 필요한 테스트가 빠진 경우
- 외부 API 또는 내부 시스템과의 호환성을 깨는 변경

## Non-blocking suggestion 기준

아래 항목은 기본적으로 `Non-blocking suggestion`으로 분류한다.

- 네이밍, 가독성, 메서드 분리, 중복 제거
- 로그 품질 개선
- 테스트 보강이 있으면 좋지만, 현재 상태만으로도 merge 차단 사유는 아닌 경우
- 장애 가능성은 낮지만 유지보수 비용이 커질 수 있는 구조
- 경계 조건 처리 방식이 다소 불명확하지만 즉시 장애로 이어질 가능성은 낮은 경우
- 문서화, 샘플 설정, 사용 예시 보강
- 이후 수정자가 이해하기 쉽도록 작은 설명, 주석, 메서드 분리가 있으면 좋은 경우

## Open question 기준

아래 항목은 기본적으로 `Open question`으로 남긴다.

- diff만으로는 의도를 단정하기 어려운 경우
- 팀 합의나 제품 정책이 필요한 경우
- 외부 의존성, 배포 환경, 권한 정책처럼 코드 밖의 정보가 필요한 경우
- 현재 변경이 의도된 제한인지 버그인지 판단이 어려운 경우
- 구현은 가능해 보이지만 왜 이런 방식으로 선택했는지 근거 확인이 필요한 경우

## priority 기본 매핑 가이드

- `Blocking issue`: 기본적으로 `P1` 또는 `P2`
- 강한 `Non-blocking suggestion`: 기본적으로 `P3`
- 약한 `Non-blocking suggestion`: 기본적으로 `P4`
- 사소한 의견: 기본적으로 `P5`
- `Open question`: 기본적으로 `P3`, 맥락에 따라 `P2` 또는 `P4`

## 라인 단위 finding 작성 원칙

- 모든 finding은 가능하면 파일 경로와 시작 라인, 종료 라인을 포함한다.
- 한 finding은 하나의 핵심 문제만 다룬다.
- 여러 파일에 걸친 문제라도 첫 번째로 수정이 필요한 파일을 대표 위치로 잡는다.
- 범위가 넓어도 인라인 코멘트 기준 anchor line 하나는 반드시 정한다.
- 파일/라인을 특정할 수 없으면 일반 요약 코멘트로만 남기되, 이유를 함께 적는다.
- 가능하면 각 finding에 `priority`를 명시해 `[P1]` ~ `[P5]` prefix가 자동으로 붙을 수 있게 한다.
- `P3`, `P4`, `Open question`도 라인 단위로 특정 가능하면 실제 GitLab 인라인 코멘트 대상으로 본다.

## 중복 방지 원칙

- 기존 discussion에 이미 같은 원인의 지적이 있으면 중복 등록하지 않는다.
- 표현만 다르고 본질이 같은 문제도 하나로 합친다.
- 같은 문제를 markdown과 GitLab 인라인 코멘트에 모두 남기되, 메시지는 서로 모순되지 않게 유지한다.

## 작성 원칙

- blocking issue는 merge 전 수정이 필요한 경우에만 사용한다.
- 확신이 낮으면 단정 대신 질문 형태를 사용한다.
- 해결 방향은 제안하되, 팀 합의가 필요한 내용은 강하게 단정하지 않는다.
- 코멘트 본문에는 토큰, 내부 URL, 민감한 로그를 포함하지 않는다.
